# Dynamic Interview Engine 設計メモ

最終更新: 2026-06-28
ステータス: 設計フェーズ（未実装）

---

## 1. 目的

現行の5問問診は「比較のための最低限の軸」を安定して取得するために存在する。
しかし店長が本当に知りたいのは、**数字の背景にある理由と文脈**である。

Dynamic Interview Engine は、5問完了後に AI が自然な会話で背景情報を補完し、
店長が面接前に「この人の事情と動機」を把握できる **採用カルテ** を生成することを目的とする。

**ゴールはスコアを上げることではない。**
店長が面接前に知りたい背景情報が増えることである。

---

## 2. なぜ現行 step machine を残すか

現行の5問問診には以下の長所がある。

| 長所 | 理由 |
|---|---|
| 壊れにくい | rule ベース。AI の気まぐれに左右されない |
| テストできる | parser 単体テスト・ペルソナテストで再現可能 |
| 速い | 5問で基本スコアが確定する |
| 比較可能 | 全応募者が同じ軸で評価される |

これを捨てると「速さと安定性」を失う。
AI深掘りは **この後に追加する** 形が最も現実的。

```
現行5問問診（rule ベース・安定）
  ↓ 基本スコア確定
AI深掘りフェーズ（OpenAI・柔軟）
  ↓ 未充填スロットを自然に補完
採用カルテ生成（構造化JSON）
  ↓
店長判断（面接前に確認）
```

## 3. 深掘り対象スロット

AI が自然な会話で埋めようとする情報の単位を「スロット」と呼ぶ。
スロットが埋まったら、その項目は二度聞かない。

### スロット定義

| スロット名 | 取得したい情報 | 発動条件の例 |
|---|---|---|
| `days_reason` | 勤務日数制限の理由（昼職・学校・育児・介護・副業） | 週3日以下を希望 |
| `curfew_reason` | 門限・帰宅時間制約の理由（終電・子供・昼職・家族） | 「〇時まで」希望がある |
| `motivation` | 志望動機（収入・経験・友人紹介・興味・独立資金） | 未聴取の場合 |
| `anxiety` | 不安要素（身バレ・お酒・接客・家族・彼氏） | 会話中に不安ワードが出た場合 |
| `future_plan` | 将来希望（副業継続・本業化・短期・長期） | 副業または掛け持ちが判明した場合 |
| `family_situation` | 家庭・育児・学校・昼職事情の詳細 | days_reason が child / school / day_job の場合 |
| `boyfriend_situation` | 交際相手の理解度・反対の有無 | anxiety が boyfriend の場合 |

### 優先度

全スロットを必ず埋めようとしない。
応募者の回答から自然に繋がるスロットだけを追う。

```
必須スロット（可能な限り取得）
  - days_reason
  - motivation

任意スロット（話が出た場合のみ深掘り）
  - curfew_reason
  - anxiety
  - future_plan
  - family_situation
  - boyfriend_situation
```

## 4. 会話設計の注意点

### 尋問感を出さない

悪い例：
```
AI: 週3日をご希望の理由を教えてください。
AI: お子さんはいますか？
AI: 彼氏は反対していますか？
```

良い例：
```
AI: 週3日をご希望とのことですが、何かご都合があれば教えてください😊
応募者: 昼の仕事をしています。
AI: そうなんですね！ 将来的には夜一本も考えていますか？それとも副業として続けたい感じでしょうか？
```

質問は **提案形・柔らかい表現** にする。
「教えてください」より「よかったら教えてください」。

### 1回に1問だけ

複数のスロットが未充填でも、一度に複数質問しない。

悪い例：
```
AI: 週3日の理由と、終電の時間と、今後のご予定を教えてください。
```

### 答えたくない内容は無理に聞かない

応募者が曖昧な答えをした場合、スロットを `null` のままにして先に進む。
「もう少し詳しく」と二度聞かない。

身バレ・家族事情・交際相手は **特に慎重に**。
話が出たから聞くのはOK。こちらから先に振るのは原則しない。

### 深掘りは最大3往復まで

基本問診5問 + AI深掘り最大3往復 = 全体で8〜10往復以内に収める。
それ以上は応募者の離脱リスクが上がる。

### 終了の宣言

深掘りが終わったら、AIは自然な言葉で締める。

```
AI: ありがとうございます😊 一通り聞けました！
   後ほど担当者からご連絡いたします。面接でお会いできるのを楽しみにしています。
```

## 5. 採用カルテ JSON 案

深掘り完了後、AIが会話全体を要約して生成する。

```json
{
  "days_reason": "day_job",
  "days_reason_detail": "昼職あり。土日休みのため金土出勤可能性あり。",
  "curfew_reason": null,
  "motivation": "income",
  "motivation_detail": "収入目的。副業として続けたい意向。",
  "anxiety": "identity",
  "anxiety_detail": "身バレが不安。職場関係者が来ないか気にしている様子。",
  "future_plan": "side_job",
  "future_plan_detail": "副業継続希望。昼職を辞める予定はない。",
  "family_situation": null,
  "boyfriend_situation": "supportive",
  "boyfriend_situation_detail": "特に反対なし。",
  "work_style": "double_work",
  "child": false,
  "school": false,
  "communication_impression": "丁寧で誠実な印象。質問への回答が具体的。",
  "continuity_estimate": "medium",
  "flag_for_interview": [
    "昼職との両立が可能かを面接で確認する",
    "身バレ対策として勤務エリアの確認が必要"
  ],
  "interview_confirmed": {
    "昼職との両立が可能かを面接で確認する": true,
    "身バレ対策として勤務エリアの確認が必要": false
  },
  "interview_additional_notes": "笑顔が自然だった。返事が早い。空気が読める。ハキハキしている。目を見て話す。",
  "interview_note_version": 1,
  "ai_summary": "昼職との副業として勤務希望。金土入れる可能性があり即戦力にはなりやすい。身バレ不安あり、丁寧なフォロー推奨。長期定着より中期安定型と見る。"
}
```

### 数値化（採用カルテから派生）

採用カルテが主役。スコアはその派生情報として計算する。

```
採用カルテ
  ↓
continuity_estimate   → 継続性スコア加算
motivation            → 動機スコア加算
days_reason           → 安定性スコア加算/減算
anxiety               → 注意フラグ
  ↓
最終スコア・Grade
```

## 7. テスト方針

### 段階1：要約精度テスト（最初にやるべき）

実際の会話ログをAIに渡して、採用カルテJSONが正しく生成されるか確認する。

```php
// scripts/test_interview_summary.php
$conversation = [
    ['role' => 'assistant', 'content' => '週3日をご希望の理由を教えてください😊'],
    ['role' => 'user',      'content' => '昼の仕事があって...'],
    // ...
];
$result = twin_interview_summarize($conversation);
assert($result['days_reason'] === 'day_job');
assert($result['work_style'] === 'double_work');
```

### 段階2：会話フローテスト

ペルソナテストと同様に、AIとの模擬会話を再現して深掘りが適切に機能するか確認する。

確認項目：
- 同じスロットを二度聞いていないか
- 3往復以内に収まっているか
- 不適切な質問（個人情報の過剰収集）が出ていないか
- 締めの文言が自然か

### 段階3：実運用テスト

テスト応募者（自分または信頼できる人）で実際の問診フローを通して確認する。
採用カルテの内容が「店長が面接前に知りたいこと」を満たしているか。

## 8. 段階的な実装ロードマップ

### Step 1（最初の実装）— 深掘り質問の提案のみ

AIに自由会話させるのではなく、**次に聞くべき質問を1つ生成する** だけ。

```
基本問診完了
↓
AI が「深掘り候補質問」を1つ生成
↓
admin_applicant.php に「面接で確認するといいかも」として表示
```

実装コスト: 小。リスク: 低。

---

### Step 2 — AI深掘りフェーズの追加（応募者との会話）

基本問診完了後に、AIが応募者と数往復の自然な会話をする。
スロット充填制御を実装する。

実装コスト: 中。リスク: 中（会話制御の精度が鍵）。

---

### Step 3 — 採用カルテ生成

会話完了後にAIが全会話を要約して採用カルテJSONを生成する。
admin_applicant.php に採用カルテセクションを追加する。

実装コスト: 中。リスク: 中（要約精度の検証が必要）。

---

### Step 4 — スコアへの反映

採用カルテの内容（継続性・動機・背景）をスコア計算に反映する。
現行スコアとの乖離を分析し、重みを調整する。

実装コスト: 小（Step 3完了後なら容易）。リスク: 低。

---

### Step 5 — 学習サイクルへの組み込み

採用カルテ + 実結果（outcome_*）を紐付けて分析する。
「育児中で週3の人は定着率が高い」などのパターンを発見する。

実装コスト: 中。リスク: 低（データ蓄積が先決）。

最初は以下を見る。LINE応募率よりも「採用判断の質」を優先する。

| 指標 | 内容 |
|---|---|
| 採用カルテ充実率 | スロット充填率（必須2項目が埋まっているか） |
| 深掘り回答率 | 深掘り質問に対して応募者が回答した割合 |
| 店長メモ削減 | 店長が面接後に追記するメモの量が減ったか |
| 面接判断しやすさ | 店長の主観評価（面接前に「この人の事情がわかった」か） |

## 設計判断メモ

- **「AIに自由を与えるほど壊れやすくなる」** — 最初は制御された深掘りAIにする。自由会話は Step 5 以降。
- **要約ステップが最大のリスク** — 会話の曖昧な発言をどう構造化するかが精度の鍵。テストを先行させる。
- **応募者体験を最優先** — 心理的な負担を感じさせない会話設計が技術と同じくらい重要。
- **採用カルテが主役、スコアは派生** — 数字よりも「人となりの理解」を先に置く。

## このエンジンは何を学習するのか

Dynamic Interview Engine の最終的な価値は、
「AIが上手に質問できるようになること」ではない。
**「優秀な店長がどんな情報を集め、どう判断しているか」をデータとして蓄積すること**である。

以下の5つが、このエンジンが積み上げていく学習対象である。

### ① どの質問をすると応募者が答えてくれるか

同じスロットを聞くにも、言い方によって回答率が変わる。
「理由を教えてください」より「よかったら教えてください」の方が答えてもらいやすいかもしれない。
回答率・離脱率・回答の具体性を蓄積することで、
**最も自然に情報を引き出せる質問文** を見つけていく。

### ② どの背景情報が採用成功につながるか

`days_reason = day_job`（昼職）の人は採用後どうなったか。
`motivation = income`（収入目的）の人は定着したか、しなかったか。
採用カルテのスロット値と `outcome_status` を紐付けることで、
**採用判断に本当に効く情報** が何かを明らかにする。

### ③ どの背景情報が定着率につながるか

「育児中で週3希望だった人は、実は定着率が高い」
「副業目的でも、将来本業化を考えている人は長続きする」
採用カルテと `outcome_retention` の相関を分析することで、
**長く働いてくれる人を見分ける背景パターン** を発見する。

### ④ どの背景情報が売上につながるか

「身バレ不安があっても、志望動機が強い人は売上が伸びる」
「昼職との掛け持ちでも、金土入れる人は呼客実績が高い」
採用カルテと `outcome_sales_rank` / `outcome_bring` を紐付けることで、
**スコアだけでは見えなかった売上予測因子** を見つける。

### ⑤ 店長が最終的に確認した項目は何か

①〜④は **応募者を学習するデータ** だが、⑤だけ性質が異なる。
⑤は **店長自身の判断基準を学習するデータ** である。

`flag_for_interview`（AIが「面接で確認してください」と提案した項目）と、
店長が `outcome_note` に実際に書き残した内容を比較する。

```
AIの提案（flag_for_interview）    店長が実際に書いたこと（outcome_note）
─────────────────────────────    ──────────────────────────────────────
・昼職との両立を確認              ・昼職は問題なし
・身バレを確認                    ・親の介護で週3だった（AIは聞いていなかった）
                                  ・接客センスが非常に良かった
```

100件のデータが溜まると、以下のような分析ができる。

**AIが毎回提案しているのに、店長が一切メモしていない項目**
→ 店長にとって重要度が低い。AIの提案から外す。

例：「身バレ不安を確認」→ 店長メモへの登場率 3% → 重要度を下げる

**店長が毎回書いているのに、AIが一度も聞いていない項目**
→ 店長が重視しているのにAIが気づいていない盲点。

例：店長メモに頻出するキーワード
```
笑顔が自然 / 空気が読める / 聞き上手 / 受け答えが丁寧 / 約束を守りそう
```
→ AIは「コミュニケーション能力」を採用カルテのスロットに持っていない
→ スロットに `communication_impression` を追加する

これにより AI は **応募者ではなく、店長の判断基準そのものを学習** し始める。

## AI・店長一致率（将来の管理画面分析）

`flag_for_interview`（AI提案）と `interview_confirmed`（店長の実確認）と
`interview_additional_notes`（店長の追加メモ）の3つを突き合わせることで、
以下の3指標を算出する。

| 指標 | 定義 | 意味 |
|---|---|---|
| **提案採用率** | AI提案のうち店長が実際に確認した割合 | AIが役に立っているか |
| **見落とし率** | 店長だけが確認した項目の割合 | AIが気づいていない盲点 |
| **不要提案率** | AIは提案したが店長は確認しなかった割合 | AIのノイズ・精度の無駄 |

```
────────────────────────────────────────────────────
AI が提案した項目数                      235 件
うち店長が確認した（提案採用率）         166 件  70%
うち店長が確認しなかった（不要提案率）    69 件  29%

店長が独自に確認した項目（見落とし率）    87 件
────────────────────────────────────────────────────
```

改善の方向が指標ごとに明確になる。

- 提案採用率が低い → AI の提案内容を見直す
- 不要提案率が高い → 不要な質問を削る（応募者の負担軽減にもなる）
- 見落とし率が高い → 店長が重視しているスロットを追加する

さらに **「AI が見落としている判断材料 TOP10」** を自動生成する。

```
店長は確認しているが AI は一度も提案していない項目

1位  接客センス・コミュニケーション力     出現率 67%
2位  家族の理解・応援があるか             出現率 41%
3位  前の職場を辞めた理由                 出現率 38%
...
```

このランキングは **AI 改善に直結する教師データ** となる。

---

### なぜこの5つが重要か — Project TWIN の本当の独自性

このエンジンは「AIが採用を決める」ためのものではない。

一般的な採用 AI は `質問 → 回答` を学習する。
Project TWIN は違う。

```
質問
  ↓
応募者の回答
  ↓
店長が何を見たか（interview_confirmed）
  ↓
店長が追加で気づいたこと（interview_additional_notes）
  ↓
店長がどう判断したか（採用・不採用・保留）
  ↓
結果どうだったか（売上・定着・呼客）
```

この **「判断プロセス全体」** を教師データにする。

最初は `応募者 → AI評価` だった。
将来は `応募者 → AI → 店長の判断 → AI → 店長の判断基準を学習` になる。

---

## 2つの学習モデル

Project TWIN が最終的に学習する対象は、1つではなく **2つ** ある。

### 応募者モデル — 「どんな人か」

```
問診回答 + 採用カルテ → 採用結果 / 売上 / 定着率
```

「育児中で週3希望の人は定着しやすい」
「副業目的でも金土入れる人は呼客実績が高い」

応募者の属性・背景・動機と、実結果の相関を学習する。

### 店長モデル — 「何を見て判断しているか」

```
flag_for_interview vs interview_confirmed vs interview_additional_notes
```

「優秀な店長は毎回コミュニケーション能力を確認している」
「売上が高いキャストを採った時、店長は必ず○○を確認していた」

店長の着眼点・判断軸・重視する情報を学習する。

この2つは **完全に別のモデル** である。

| | 応募者モデル | 店長モデル |
|---|---|---|
| 学習対象 | 応募者の属性・背景・動機 | 店長の確認項目・判断基準 |
| 主なデータ源 | 採用カルテ JSON + outcome_* | flag_for_interview + interview_confirmed + interview_additional_notes |
| 将来の活用 | 採用スコアの精度向上 | AI提案の精度向上・盲点の発見 |

**店長そのものが教師データになる。** これが Project TWIN の最大の独自性である。

```
店長 面接
  ↓
interview_confirmed（実際に確認した項目）
  +
interview_additional_notes（笑顔が自然・返事が早い…）
  ↓
採用
  ↓
3か月後 outcome_* 記録
  ↓
AI 学習
```

この流れ全体が途切れなく記録されることが、Project TWIN の設計の核心である。

## 採用データの三層構造

採用に関わるデータは3つのレイヤーに分けて保存する。

### Layer 1 — 構造化（AIが使いやすいデータ）

```
days_reason / motivation / anxiety / future_plan /
continuity_estimate / flag_for_interview / interview_confirmed
```

- SQL 検索・集計・パターン分析に使う
- Grade / スコア計算の入力になる
- **今すぐ使える**

### Layer 2 — 人間の言葉（店長の知識）

```
interview_additional_notes  TEXT

例:
笑顔が自然 / 返事が早い / 空気が読める / ハキハキしている /
気遣いできそう / 素直そう / 約束守りそう / 目を見て話す /
受け答えはぎこちないけど誠実そう / 押しが弱い
```

- 一切変換しない。店長の言葉をそのまま保存する
- 将来 Embedding 化して意味的な近さを学習する
- **「笑顔が自然」を `communication: "high"` に変えた瞬間、「ぎこちないけど誠実」が表現できなくなる。構造化は後から AI がやればいい**
- **データが 50 件以上溜まってから活用開始**

```
笑顔が自然     →┐
素直そう       →┤ Embedding → 「接客適性」クラスタ
受け答えが丁寧 →┤
空気が読める   →┘
```

### Layer 3 — 結果（検証用データ）

```
outcome_status / outcome_sales_rank /
outcome_retention / outcome_bring
```

- すでに実装済み（migration 010）
- Layer 1 + Layer 2 と紐付けることで、「どんな感覚が結果につながったか」をはじめて学習できる

### なぜ三層か

```
構造化（Layer 1）
  ↓ 紐付け
人間の感覚（Layer 2）
  ↓ 紐付け
結果（Layer 3）
```

この3つが揃ったとき、AI は初めて「どんな感覚が結果につながったか」を学習できる。

| | Layer 1 | Layer 2 | Layer 3 |
|---|---|---|---|
| 内容 | 構造化スロット | 店長の生言葉 | 採用・売上・定着 |
| 強み | 集計・比較が容易 | 情報が失われない | 正解ラベルになる |
| 弱み | 定義外の情報が落ちる | 単純集計できない | 3か月後まで揃わない |
| 活用開始時期 | 今すぐ | 50件以上から | 採用後3か月後 |

## バージョン管理 — 未来の AI が学習しやすいようにデータを残す

採用カルテには3つのバージョンフィールドを持たせる。
目的は「このカルテがどの時代に、どのスキーマで、何が生成したか」を永続的に記録することである。

```json
"interview_note_version": 1,
"slot_schema_version": 1,
"ai_model_version": "gpt-4o-2024-11-20"
```

### `interview_note_version` — 店長メモの形式

```
v1  自由記述のみ（現在）
v2  AI 自動タグを追加（将来）
v3  Embedding インデックス化（さらに将来）
```

将来 AI が自動タグを生成する `v2` に移行しても、過去データが `v1` の自由記述であることを識別できる。

### `slot_schema_version` — スロット定義のバージョン

```
v1  7スロット（現在）
    days_reason / curfew_reason / motivation / anxiety /
    future_plan / family_situation / boyfriend_situation

v2  10スロット（将来）
    + communication_impression / work_ethic / interview_impression など

v3  15スロット（さらに将来）
    店長メモの分析から発見されたスロットを追加
```

分析時に「この時代は7スロットだった」が分かる。
スキーマが変わっても過去データとの比較が可能になる。

### `ai_model_version` — 採用カルテを生成した AI モデル

```
gpt-4o-2024-11-20  → 現在使用
gpt-5.x            → 将来移行
独自ファインチューニングモデル → さらに将来
```

モデルが変わると、同じ会話から異なる要約が生成される可能性がある。
どのモデルが生成したカルテかを記録しておくことで、
「GPT-4o 時代のカルテ」と「GPT-5 時代のカルテ」の品質差を比較・検証できる。

### なぜこの3つが重要か

```
今やること           未来の AI がやること
────────────────    ────────────────────────────
事実を壊さず保存      パターンを発見する
言葉を変換しない      意味を学習する
バージョンを記録する   過去データを正しく解釈する
```

**「データを集める」と「未来の AI が学習しやすい形でデータを残す」は全然違う。**

前者は今のツールで読めればいい。
後者は5年後のモデルが過去データを遡って学習できる状態を維持する必要がある。

Project TWIN の設計思想は後者である。

## Project TWIN が本当に作っているもの

最初の定義は「夜業界の運営OS」だった。
それは正しい。

しかしデータ設計の観点から見ると、より本質的な表現がある。

> **Project TWIN は、「人間の優れた判断を、未来の AI が学習できる形で保存するシステム」である。**

だから、

- 応募者の事実は壊さず保存する
- 店長の言葉も壊さず保存する
- 結果もそのまま保存する
- 解釈・Embedding・分類は、未来のより賢い AI に任せる

「保存を優先し、解釈を急がない」という方針は、AI の性能が年々向上することを前提にすると合理的である。
5年後のモデルは、今日では気づけないパターンを見つけられるかもしれない。その可能性を閉じない設計になっている。

Project TWIN が積み上げるのは採用データではなく、
**優秀な店長が何を見て、どう判断し、どんな結果を出したか** という知見そのものである。
これは単なる採用ツールでは蓄積できない、Project TWIN だけが持てる競争優位性になる。
もし全国に展開されれば、「売れる店長は何を見て採用しているか」という知見が、
個人の経験ではなくデータとして業界に蓄積される。
