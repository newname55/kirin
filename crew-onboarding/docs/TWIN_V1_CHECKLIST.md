# Project TWIN v1.0 公開前チェックリスト

作成日: 2026-06-23

## 目的

Project TWIN を外部サーバーへ移設・公開する前に、複数店舗対応、DB 再現性、管理画面の安全性、OpenAI 障害時の挙動、ログと個人情報の扱いを確認する。

このドキュメントは v1.0 公開可否を判断するための運用チェックリストであり、実装変更は含まない。

## 前提

- 対象店舗は `seika` / `creole`。
- 外部サーバーでは `TWIN_STORE_KEY` または `app/config.local.php` の `store_key` で店舗を切り替える。
- 店舗サイト本体と Project TWIN は別 Git 管理とする。
- `app/config.local.php`、API キー、DB パスワード、サーバー固有設定は Git に含めない。

## 1. 店舗別設定

### 目的

星華固定の実装を残さず、1つのコードベースで複数店舗へ展開できる状態にする。

### 現状確認ポイント

- `app/store.php` で `store_key` が解決されているか。
- `app/knowledge/stores.php` に `seika` / `creole` の店舗設定が揃っているか。
- `openai_engine.php` / `rule_engine.php` / `public/index.php` / `public/admin.php` が現在店舗の設定を参照しているか。
- `Club 星華`、`seika`、星華URL、星華LINE、星華Instagram、星華料金URLの直書きが残っていないか。
- 管理画面の店舗切替で、店舗ナレッジ、AIキャラクター、CTAリンクが正しく切り替わるか。

### 実装・修正が必要な内容

- `creole` の正式店舗情報を `stores.php` に入力する。
- 店舗追加時に必要な設定項目を README または SYSTEM_SPEC に追記する。
- 星華固定の文言・URL・WBSS store key が残っている場合は `twin_store_value()` または店舗設定参照へ置き換える。
- 管理画面の選択店舗と公開チャットの `store_key` 解決順序を明文化する。

### 完了条件

- `TWIN_STORE_KEY=seika` / `TWIN_STORE_KEY=creole` の両方でチャット画面と管理画面が店舗別表示になる。
- `rg "Club 星華|星華|seika"` の結果を確認し、固定で残すべき情報と置き換えるべき情報が分類済み。
- 新店舗追加時に `stores.php` と環境設定だけで最低限公開できる。

### リスク

- 星華固定のURLや料金が creole 公開画面に出ると、誤案内・予約導線ミスにつながる。
- 店舗別設定が不足すると、OpenAI 応答だけでなく rule 応答も誤る。
- 管理画面の店舗切替と公開チャットの店舗設定がズレると、運用者が誤った店舗情報を編集する。

### 優先度

高

## 2. DBマイグレーション整理

### 目的

DB を別サーバーへ移しても、同じスキーマを再現できる状態にする。

### 現状確認ポイント

- `database/schema.sql` が新規構築時の最新スキーマとして使えるか。
- `database/migrations/` の実行順序が明確か。
- v0.9 系で必要な `ai_character_settings`、`app_settings`、`ai_usage_logs`、会話分析系カラムが再現できるか。
- `messages.store_key` または同等の店舗別ログ識別が未実装であることを公開前の課題として扱っているか。
- 既存 DB から移行する場合と新規 DB を作る場合の手順が分かれているか。

### 実装・修正が必要な内容

- 新規構築用の `schema.sql` と差分適用用 migration の関係を整理する。
- migration 実行順を docs に固定化する。
- `messages.store_key` 追加方針を決める。候補は `chat_messages.store_key`、`chat_sessions.store_key`、または両方。
- 店舗別分析を行うなら、`event_logs` と `chat_messages` のどちらに `store_key` を持たせるかを決める。
- migration 実行後の検証SQLを用意する。

### 完了条件

- 空の DB に対して、手順どおりに schema / migration を適用すれば管理画面とチャットが動く。
- 既存 DB に対して、重複実行や途中失敗時の復旧方針が明記されている。
- `show tables` と主要カラム確認で v1.0 に必要なテーブル・カラムが揃う。
- DB バックアップ、migration 適用、rollback の順序が運用手順に書かれている。

### リスク

- migration 手順が曖昧だと、Xserver 側で管理画面だけ落ちる、AIキャラクター設定だけ保存できないなど部分障害になる。
- 店舗IDなしでログを保存し続けると、複数店舗展開後に分析が混ざる。
- rollback 方針がないまま本番 DB を変更すると、公開後の復旧に時間がかかる。

### 優先度

高

## 3. 管理画面の権限

### 目的

管理画面が外部公開されても、第三者に設定変更・ログ閲覧・API診断情報閲覧を許さない状態にする。

### 現状確認ポイント

- `public/admin.php` がセッションログインで保護されているか。
- 管理者ID・パスワードがコードに直書きされていないか。
- `config.local.php` または環境変数で管理者認証情報を設定できるか。
- CSRF 対策が設定変更フォームに適用されているか。
- `admin_export.php` が未ログイン状態で会話ログや分析JSONを出さないか。
- `X-Robots-Tag: noindex, nofollow` と Basic 認証など、外部公開時の追加防御があるか。

### 実装・修正が必要な内容

- 公開サーバーでは強い管理者パスワードへ変更する。
- 可能ならサーバー側 Basic 認証、IP制限、または管理画面URLの限定公開を追加する。
- セッションCookieに `Secure` / `HttpOnly` / `SameSite` を設定する。
- ログエクスポートは管理者のみ、かつ必要最小限の期間で出力する。
- Dev タブ、OpenAI診断、AI利用額など外部公開時に不要な情報は表示条件を設ける。

### 完了条件

- 未ログインで `admin.php`、`admin_export.php`、設定変更APIにアクセスしても情報が出ない。
- 管理者認証情報が Git 管理外で設定されている。
- HTTPS 環境でセッションCookie属性が適切に設定されている。
- 公開URLを知っているだけでは管理画面に入れない。

### リスク

- 管理画面が突破されると、店舗情報改ざん、CTA差し替え、会話ログ閲覧が可能になる。
- エクスポート機能から会話ログが漏れると、個人情報を入力した利用者がいた場合に二次被害につながる。
- Dev 情報を公開したままにすると、API設定や障害状況を外部に推測される。

### 優先度

最高

## 4. エラーログ

### 目的

公開後の障害原因を追える一方で、サーバー移設時にもログ書き込みエラーでアプリが止まらない状態にする。

### 現状確認ポイント

- `storage/logs/` が公開サーバーで書き込み可能か。
- PHP の `error_log` 出力先が確認できるか。
- OpenAI、WBSS、DB、アップロード、CSRF、認証失敗のログが追えるか。
- ログに API キー、DB パスワード、個人情報が出力されないか。
- ログローテーションまたは定期削除方針があるか。

### 実装・修正が必要な内容

- `storage/logs/` と `public/uploads/` の権限設定をデプロイ手順に追加する。
- 本番では詳細エラーを画面表示せず、利用者には自然な案内を返す。
- ログ出力時に秘密情報をマスクする。
- Xserver 側の PHP エラーログ確認方法を docs に追記する。
- ログ保存期間と削除手順を決める。

### 完了条件

- 書き込み権限不足、DB接続失敗、OpenAI失敗、WBSS失敗をそれぞれ再現し、ログで追える。
- 利用者画面にスタックトレースやパスが表示されない。
- ログに API キー、電話番号、LINE ID、住所、決済情報などが残らない。
- ログ容量が増え続けるリスクへの運用方針がある。

### リスク

- ログが書けないと障害調査が遅れる。
- 詳細エラーを画面に出すと、ファイルパスや内部構造が漏れる。
- ログに個人情報や秘密情報が混ざると、管理画面以外からの漏えい経路になる。

### 優先度

高

## 5. OpenAI失敗時の挙動

### 目的

OpenAI API が未設定、タイムアウト、エラー、レート制限になっても、ユーザーに不自然なエラーを出さず、予約導線を維持する。

### 現状確認ポイント

- `RESPONSE_MODE=openai` / `hybrid` で OpenAI 失敗時に rule engine へフォールバックするか。
- `fallback_rule` がログに残るか。
- ユーザー向け文言が技術的すぎないか。
- 料金、場所、営業時間、LINE、Instagram、出勤など重要情報は OpenAI なしでも返せるか。
- OpenAI API Key 未設定時の挙動が公開前に確認されているか。

### 実装・修正が必要な内容

- OpenAI API Key 未設定、401、429、500、タイムアウトの各ケースをテストする。
- 失敗時の案内文を店舗トーンに合わせて調整する。
- `hybrid` モードで rule が担当する intent と OpenAI が担当する intent を最終確認する。
- OpenAI 失敗時も LINE 予約・料金・Instagram の CTA が表示されることを確認する。
- 管理画面の OpenAI 診断で失敗件数・最終エラーを確認できる状態にする。

### 完了条件

- OpenAI API を意図的に失敗させても、チャット画面が 500 にならない。
- ユーザーには「ただいま確認しながらご案内します」など自然な案内が出る。
- 重要 intent は rule 応答または固定CTAで来店導線が維持される。
- 管理画面に `openai_error` / `openai_fallback` が記録される。

### リスク

- OpenAI 障害時に白画面や技術エラーが出ると、公開直後の信頼を落とす。
- fallback が自然でないと、会話品質が急に下がる。
- OpenAI に推測させる範囲が広すぎると、料金・出勤・店舗情報の誤案内につながる。

### 優先度

高

## 6. 個人情報を保存しない設計確認

### 目的

Project TWIN は来店導線改善のための会話・行動ログを扱うが、本名、電話番号、LINE ID、住所、決済情報、身分証情報などの個人情報を保存しない方針を明確にする。

### 現状確認ポイント

- 会話ログにユーザー入力全文を保存しているため、利用者が個人情報を入力した場合の扱いが決まっているか。
- 管理画面・CSV・JSON エクスポートに会話内容が含まれることを運用者が理解しているか。
- 個人情報らしい入力を検知した場合に保存しない、マスクする、または削除できる設計があるか。
- 保存してよいデータと保存しないデータが明文化されているか。
- 外部公開前にプライバシーポリシーや利用者向け注意文言が必要か判断しているか。

### 実装・修正が必要な内容

- 保存してよいデータを次に限定する。
  - 会話内容
  - intent
  - CTAクリック
  - 日時
  - 店舗ID
  - 応答モード
  - エラー種別
- 保存しないデータを明記し、検知・削除・マスク方針を決める。
  - 本名
  - 電話番号
  - LINE ID
  - 住所
  - 決済情報
  - 身分証情報
- 会話ログ保存前に個人情報パターンをマスクするか、管理画面から削除できる運用を追加する。
- CSV / JSON エクスポートの取り扱い注意を管理画面と docs に明記する。
- ログ保存期間と削除依頼時の対応手順を決める。

### 完了条件

- 保存対象・保存禁止対象が docs に明記されている。
- 本名、電話番号、LINE ID、住所、決済情報、身分証情報を保存しない方針が運用者に共有されている。
- 個人情報が誤入力された場合の削除手順またはマスク手順がある。
- エクスポートデータを管理者以外に渡さない運用になっている。

### リスク

- ユーザーが自由入力欄に個人情報を入れる可能性がある。
- 会話ログをそのまま長期保存・エクスポートすると、個人情報混入時の影響が大きい。
- 店舗別運用が増えるほど、ログ閲覧権限と削除責任が曖昧になりやすい。

### 優先度

最高

## 公開判定

- `最高` 優先度の項目が1つでも未完了の場合、v1.0 公開不可。
- `高` 優先度の項目に未完了がある場合、公開日を決める前に対応担当と期限を決める。
- `中` 以下の改善は公開後対応でもよいが、外部公開時のセキュリティ・個人情報・誤案内に関わるものは公開前に完了する。
